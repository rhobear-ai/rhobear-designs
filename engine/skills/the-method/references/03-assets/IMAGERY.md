# IMAGERY — sourcing pictures

Imagery is where the "AI made this" tell is loudest, because it is where a blank is cheapest to
fill. A brief that does not say what the picture is gets a picture of nothing in particular:
technically competent, tonally generic, and interchangeable with every other site's.

> **Every image in a project has a declared tier and a declared job. An image with no job does not
> ship.**

---

## The generated picture is the foundation, not a tier among tiers

Read the tiers below as **shipping rules** — what may leave the building as a final asset. They do
**not** describe where the design *starts*. The design starts at the picture.

> **The method is image-first. The picture comes first, the CSS is built to match the picture, then
> the JS/motion. Not the other way around.** You do not write tokens and then illustrate them; you
> generate the frame, approve it, and the approved frame becomes the specification the code is held
> to (Stage 8b). Every product in the worked run — home page then eight apps — was designed from a
> generated frame, not from a token sheet.

That makes the **image generator the spine of the whole pipeline**, not an optional garnish in a
"generators/" folder. Here that generator is **Gemini Nano Banana (Pro)**, reached through the
Vertex gateway (the $1k credit) and the house Hermes gateway. It is the foundation for two reasons,
and they are different jobs:

1. **As the design tool** — it produces the mocks this system is built to scrub and codify. Miss
   this framing and the pipeline reads as "write CSS, maybe add a picture," which is backwards and
   is exactly the drift that flattens the output.
2. **As a shipped product feature** — the engine this system becomes (`OpenDesign` fork → sever)
   **pipes Nano Banana in and generates instantly**, in-product. The user prompts it and it makes
   the frame, then builds the system around it. The output carries no "Nano Banana" branding — at
   most a "generated by" line on download. The generation *is* the product, not a preprocessing step.

None of this contradicts the shipping rule that a generated **UI comp is a spec, not the built
deliverable** (a comp with mangled text must never reach a customer). "Foundation of the design" and
"reference for the build" are the same picture wearing two hats: it leads the process **and** it is
held to, and rebuilt as, real code before anything ships.

---

## The four tiers

| Tier | What it is | Ships as | Rule |
|---|---|---|---|
| **Owned** | commissioned, shot, or bought outright — marks, illustration, product photography | vector or master file | the brand. Never regenerated, never restyled. |
| **Licensed** | stock with a paid licence — a reference bear, a texture, a plate | as licensed | keep the licence with the asset |
| **Generated** | model output — UI comps, storyboards, environment plates, textures | **reference**, mostly | see below |
| **Captured** | screenshots of the real running product | PNG in the ux-pack | the fact-check for everything else |

---

## Generated imagery — reference vs. shipped

The distinction that keeps a project honest:

**Generated images are reference by default.**

A generated UI comp is a *specification for a build*, not the build. A generated mark is a
*direction for a vector artist*, not a logo. Shipping a generated raster where a vector belongs is
how a brand ends up with a mark that cannot be resized, recoloured, or printed — and the failure
surfaces at the worst moment, usually at production.

**Generated images may ship when:**

- they are **environment or texture plates** consumed by a scene (terrain albedo, normal maps, sky)
- they are **background art** with no type and no mark in them
- they went through a **vector pass** — traced, cleaned, and redrawn by a human, with the original
  as reference

**Generated images never ship when:**

- they contain **text** — generators mangle it, and the mangling is invisible until it is in front
  of a customer
- they contain the **mark** — the mark comes from the pack, always
- they are a **UI comp** — the comp is the spec; the build is the deliverable
- they depict a **real person, place, or product** you do not have rights to

## Sourcing environment and texture plates

For scene work, the plate prompt is a **material description**, not a scene description. You are
asking for a surface, not a picture:

```
Seamless PBR terrain texture, <condition>, <materials>, <detail>,
top-down orthographic view, 4K tileable, no shadows, flat lighting
```

`no shadows, flat lighting` is doing critical work. A plate with baked lighting fights the scene's
own light rig and cannot be relit, so it will look wrong at every beat except the one it was
generated for.

Generate the **normal map as a separate pass** with the same surface description. Deriving a
normal map from an albedo by height-guessing produces mush at grazing angles, which is exactly
where a scroll-scene camera spends its time.

Three plates for a scroll-scene — start, middle, end of the environment gradient — blended by one
`progress` uniform. Not one per beat: the blend is what makes it continuous.

## Licensing — keep it with the asset

Every licensed and every owned asset carries its licence alongside it:

```
assets/
  marks/
    grizzly-pack/
      LICENCE.md          ← what was bought, from whom, what it permits
      hero.svg
      expressive.svg
      compact.svg
      micro.svg
  textures/
    terrain-winter/
      LICENCE.md
      albedo-4k.png
      normal-4k.png
```

`LICENCE.md` states: source, date, licence type, **what it permits** (web / print / product /
resale / modification), and what it does not.

The question always arrives later — at a print run, an app-store submission, or a resale — and by
then whoever bought it has moved on. A licence recorded next to the file survives the person; a
licence in someone's inbox does not.

**Never:** an image scraped from a site, a "royalty-free" file with no traceable source, a mark
that resembles an existing brand's, or a real person's likeness without a release.

---

## The declaration, in DESIGN.md

```yaml
imagery:
  owned:     [<marks, illustration, product photography>]
  licensed:  [<stock, with source and licence>]
  generated: [<what may be generated, and whether reference or shipped>]
  captured:  <path to the ux-pack>
  never:     [<explicit exclusions — real people, competitor products, etc.>]
```

The `never` list matters more than it looks. It is the only place a project states what it will
not do, and an agent with no exclusion list treats everything as permitted.

## Anti-default

| Banned | Why |
|---|---|
| Generic stock — smiling team at a laptop, abstract network nodes, handshake | the visual equivalent of lorem ipsum |
| Purple-blue gradient mesh backgrounds | the single most recognisable AI-era default |
| An image with no job, added for balance | balance is a layout problem; solve it in layout |
| Generated text inside an image | mangled, and invisible until a customer sees it |
| A generated mark shipped as the mark | cannot be resized, recoloured, or printed |
| Baked lighting in a texture plate | fights the scene rig; wrong at every beat but one |
| Mixed illustration styles in one product | reads as assembled from parts, because it was |
| Photography and illustration at the same weight | pick which one carries the brand |

---

## Checklist

```
[ ] every image has a declared tier and a stated job
[ ] no generated image ships with text or the mark in it
[ ] every licensed and owned asset has LICENCE.md beside it
[ ] the never-list is written in DESIGN.md
[ ] texture plates are flat-lit and tileable, with separate normal passes
[ ] one illustration style across the product
[ ] every generated image opened and observed before approval
```

The last line is the same rule as everywhere else in this system: **nothing ships unread.** An
image approved without being opened is the cheapest possible way to put mangled text, a wrong
mark, or a hallucinated product name in front of a customer.
